# Companion Hub 活动数据刷新设计

## 目标

修复 Companion Hub 的“今日专注”“今日完成”和“最近项目”在有效 Git 活动产生后不能自动更新的问题。三项数据必须来自同一份可信活动快照，在页面重建、应用前后台切换、跨日及应用重启后保持一致；不得通过硬编码、轮询或单纯强制重绘掩盖数据链路问题。

## 已确认的数据链路与问题

`script/update_git_completion_count.sh` 扫描已授权 Git 根目录，生成当天专注块、提交数与最近项目，并写入 App Group 中的可信快照。`GitTodayActivityStore` 读取可信快照或兼容性 Store，`GitActivityRefreshCoordinator` 在启动、前台恢复、授权变化和系统唤醒等事件后执行脚本并写入 `TinyBuddyCombinedSnapshotStore`。`PetViewModel` 将组合快照发布为 `@Published` 展示状态，`PetView` 负责呈现。

当前链路存在四处一致性缺口：

1. `GitTodayActivityStore` 使用不可注入的 `Calendar.current` 与 `Date()` 判断当天，而下层 Store 可以使用固定日期和日历，导致跨午夜或时区变化时快照有效性判断不一致。
2. 刷新成功没有独立的“活动快照已提交”状态事件。`PetViewModel` 主要依赖刷新状态或应用激活通知重载，因此脚本成功写入后不能保证立即发布新数据。
3. 刷新成功条件仅保证专注数和提交数存在，没有将最近项目纳入同一原子快照语义，可能在统计有效时丢失或错误清空项目。
4. 现有测试分别覆盖 Store、协调器与 ViewModel，但缺少从数据写入到 UI 状态发布及恢复的一致性回归场景。

## 采用方案

保留现有脚本、可信快照、组合快照和 ViewModel 分层，不引入新轮询或大型状态框架。修复各层之间的契约：统一日期上下文、原子提交完整活动切片、在提交成功后发布专用变更事件，并让 ViewModel 以持久化组合快照为唯一恢复基准。

相比只增加通知，该方案同时覆盖跨日、最近项目和重启恢复；相比重建全局状态容器，改动范围更小并沿用现有架构。

## 详细设计

### 统一日期聚合

为 `GitTodayActivityStore` 注入与下层 Store 相同的日期提供器和日历，并由单一函数生成 day identifier。可信快照、兼容 Store 回退和组合快照读取均使用同一个“当前日”上下文。

日期变化后，昨日快照不得作为今日非零数据返回。当天首次有效刷新将原子写入新 day identifier；刷新前应呈现明确的今日空状态，而不是昨日统计。

### 原子活动切片

协调器只从一次 `GitTodayActivityStore` 读取结果构造活动切片。切片包含 day identifier、专注数、提交数和可选最近项目，随后一次性提交到 `TinyBuddyCombinedSnapshotStore`。

“最近项目”为空是合法状态，但必须是显式的 `nil`/空状态语义，不能因为读取失败、半写入或兼容键缺失而覆盖一份有效项目。脚本或可信快照不可用时保留上一份同日已确认数据，并走现有失败路径；不得把失败解释成零值成功。

### 状态发布与界面刷新

组合快照成功提交后，协调器在主线程发布专用活动快照变更通知。通知表示持久化提交已完成，而不是仅表示脚本结束或 Widget 重载。

`PetViewModel` 监听该通知，并调用现有快照装载与发布路径更新 `hudPresentation`、统计和显示状态。SwiftUI 继续通过 `@StateObject`/`@Published` 自然刷新；不增加定时轮询、视图 identity 切换或无条件强制重绘。

页面重建从组合快照立即恢复；应用回到前台时现有生命周期刷新继续负责发现外部活动，成功提交后由同一通知发布。应用重启时先显示持久化的同日快照，再由启动刷新校正。

### 持久化与兼容性

可信活动快照仍是真实 Git 活动输入，组合快照仍是 app/widget 的一致展示状态。保留兼容性 Store 回退，但所有回退都必须通过统一日期校验。写入成功后继续镜像必要的兼容键，避免现有 Widget 或旧版本升级路径失效。

不修改授权根目录、App Group、签名或脚本扫描规则。

### 错误处理

- 脚本失败、可信载荷无效或活动计数缺失：不发布成功变更事件，不覆盖有效同日快照。
- 最近项目合法为空：发布包含空项目的完整快照，界面显示既有空状态。
- 组合快照提交失败：保持先前已发布状态并记录现有遥测，不发送误导性成功通知。
- 跨日但尚无今日活动：返回今日零/空状态，不泄漏昨日项目或计数。

## 回归测试

在 Core 与 App 测试中补充以下确定性用例：

1. 数据写入：可信快照写入非零专注数、提交数和最近项目后，活动 Store 与组合快照返回同一完整切片。
2. 刷新触发：协调器成功提交后发送一次活动变更事件；`PetViewModel` 无需 Widget reload、重启或前后台切换即可更新三项展示值。
3. 跨日边界：使用固定日历和可变日期提供器验证午夜前后；昨日非零数据不会出现在今日，今日写入后正确恢复。
4. 空状态：无活动及合法空最近项目时返回零/空展示；失败读取不覆盖有效同日值。
5. 页面重建：新建 `PetViewModel` 时立即加载同日持久化组合快照。
6. 前后台切换：激活触发刷新后，通过提交事件自动发布最新活动。
7. 持久化恢复：重建 Store/ViewModel 模拟应用重启，三项值保持一致，并遵守 revision 与 day identifier。

测试使用隔离的 `UserDefaults`、固定日历、可控日期提供器、脚本/进程 stub 和独立通知中心，避免依赖真实时间、真实 Git 仓库或轮询。

## 验证门禁

实现后先运行受影响的 `TinyBuddyCoreTests` 与 `TinyBuddyAppTests` 定向测试，再运行一次 `swift test`。若改动仅涉及 Swift 数据与状态链路，不触碰签名、Widget 注册或启动脚本，则不要求发布签名验证；如实际修改扩展到这些范围，再运行最小相关 `build_and_run.sh` 验证模式。

完成标准是：所有新增回归测试通过；现有测试无回归；代码检查确认没有硬编码统计、定时轮询、视图 identity 技巧或仅强制重绘；最终 diff 仅包含本目标所需的数据、发布链路和测试改动。
